# Backup plan — политика, расписание, восстановление

## 1) Цели

- Быстрое восстановление критичных сервисов (RTO).
- Защита данных (RPO) и понятный “runbook” восстановления.
- Разделение: бэкап VM и бэкап данных внутри storage.

## 2) Что бэкапим

### Proxmox level (VM backups)

- `vm-ai-core-01` — **обязательно**
- `vm-web-host-01` — **обязательно**
- `vm-win-work-01` — опционально (чаще: системный образ реже + данные отдельно)

### Data level (storage backups)

- `/data/projects` — обязательно
- `/data/docs` — обязательно
- `/data/shared` — по необходимости
- `/data/media` — по желанию (объём большой, можно “best effort”)
- `/data/inbox` — можно не бэкапить (если это временный буфер), либо короткое хранение

## 3) Расписание (минимальный стандарт)

- **Ежедневно (nightly)**:
  - Proxmox backup: `vm-web-host-01`, `vm-ai-core-01`
  - Data backup: `/data/projects`, `/data/docs`
- **Еженедельно**:
  - полный backup VM + проверка восстановления (минимум 1 VM)
  - `/data/shared` (если используется)
- **Ежемесячно**:
  - cold backup (offsite/внешний диск) критичных данных

## 4) Приоритет восстановления (что поднимать первым)

1. **Storage layer** (чтобы сервисы увидели данные)
2. `vm-web-host-01` (reverse proxy + критичные сервисы)
3. `vm-ai-core-01` (автоматизации/интеграции)
4. `vm-win-work-01` (рабочая среда/1C)

## 5) Что НЕ бэкапить в git

- Пароли, токены, ключи (в том числе TLS private keys)
- Конфиги VPN с приватными ключами
- Файлы базы данных “как есть” без понимания консистентности (для БД нужен корректный dump/снэпшот)
